Design Document: Antigravity Backend (Phase 3)Project: AntigravityScope: Doctor’s Consultation & Integrated Chemist WorkflowStack: Node.js, Express, Neon DB (PostgreSQL)1. Extension of Phase 1: The Chemist RegistryTo ensure every hospital has its own dedicated pharmacy, we are adding a "Chemist Entity" linked to the Hospital ID.The "Hospital-Owned Chemist" LogicDuring hospital registration (Phase 1), a new mandatory requirement is added:Chemist Shop Name: (e.g., "City Hospital Pharmacy").Staff Credentials: A separate login for the pharmacy staff to access the "Medicine Queue."2. The Doctor’s Window (Consultation Logic)This is the workspace for the doctor while the patient is physically present.A. Real-time Patient Card RetrievalWhen the doctor calls a Token Number (e.g., Number 24), the backend performs a JOIN query to fetch:From Patient Registry: Full Name, Phone, Age, Blood Group, Allergies, Chronic Diseases.From Appointment Booking: The "Problem Description" the patient wrote when they booked.Visualized: The doctor sees a complete medical summary before the patient even speaks.B. Digital Prescription & Message RoutingInstead of a paper slip, the doctor uses a specialized Prescription Text Box.Logic: Once the doctor clicks "Send to Chemist," the backend creates a Prescription record linked to that specific Appointment_ID.Payload: Contains the text (medicines/tests) and the Patient Token Number.3. The Chemist’s Window (Fulfillment Logic)The pharmacy staff sees a live "To-Pack" list.A. Preparation WorkflowIncoming Message: The staff receives a notification: "New Prescription for Token #24."Pre-Packing: Staff reads the doctor's notes and prepares the envelope with medicines beforehand.The "Safety Check" (Anti-Scam Logic): * To prevent unauthorized pickups, the Chemist's screen displays only the Token Number and Status (Ready).When a person arrives, the staff clicks on the Token Number to reveal the Full Name and Phone Number.The Rule: The staff must ask the patient for their name and verify it against the screen before handing over the envelope.4. Updated Database Schema (Neon DB)TableNew Columns / PurposeChemistsid, hospital_id, shop_name, staff_password_hashPrescriptionsid, appointment_id, doctor_id, chemist_id, content (text), status (Pending/Ready/Delivered)5. API Endpoints (Phase 3)MethodEndpointDescriptionGET/api/v1/doctor/patient-card/:tokenFetches all registration + appointment data for the doctor.POST/api/v1/doctor/prescribeSends the text box data to the database and notifies the chemist.GET/api/v1/chemist/queueFetches all "Pending" and "Ready" prescriptions for that hospital.PATCH/api/v1/chemist/verify/:idMarks medicine as "Delivered" after name/phone verification.6. Logic Summary for AntigravityThe Workflow: Doctor writes → Message triggers Chemist → Chemist packs → Patient arrives → Verification (Name/Phone) → Handover.Safety: By only showing Name/Phone to the chemist after the patient provides the Token Number, we minimize the risk of scammers claiming random envelopes.Efficiency: The "packing beforehand" logic ensures the patient doesn't wait in a second queue at the pharmacy.
