Design Document: Antigravity Backend (Phase 2)Project: AntigravityScope: Patient Discovery & Appointment LifecycleStack: Node.js, Express, Neon DB (PostgreSQL)1. Advanced Search Logic: Keywords to SpecialistsThe core requirement is that a user shouldn't need to know medical terminology. If they type "pancreas," the system must suggest Gastroenterologists.A. Symptom-to-Specialist MappingWe will create a lookup table called Symptom_Mapping.Keywords: pancreas, digestion, stomach ache, heartburn.Specialization: Gastroenterologist.B. Search Query Logic (PostgreSQL FTS)We won't use simple LIKE %query% because it's slow and dumb. We will use tsvector and tsquery in Neon.Flow: User types "pancreas" → System finds "Gastroenterologist" from mapping → System queries Doctors table for that specialty → Results returned.Doctor Search: If the user searches a name directly (e.g., "Dr. Sharma"), the query checks the full_name column in the Doctors table and returns their associated Hospital_ID.2. Appointment & Slot ManagementTo prevent double-booking and manage "favourite slots," we use a Slot-Based Availability model.A. The "Is Doctor Free?" LogicThe backend will calculate availability by checking the Appointments table for a specific Doctor_ID on a specific Date.Available Slots: 10:00 AM, 10:30 AM, 11:00 AM...Logic: If a slot is present in the Appointments table for that doctor/date, it is marked "Booked" in the UI.B. Token Number GenerationTo avoid the "missing number" problem and handle concurrent bookings (two people clicking book at the exact same second), we use a PostgreSQL Transaction.Request: Patient clicks "Book".Atomic Transaction: * SELECT MAX(token_number) FROM Appointments WHERE doctor_id = X AND date = Y FOR UPDATE;New Token = Max + 1.INSERT the appointment.Result: Patient 23 is followed by 24, guaranteed.3. Data Flow & "The Token Card"Once booked, the system generates a JSON object (The "Card") containing the joined data:From Patient Profile: Blood Group, Diseases, Allergies.From Current Booking: Token Number, Time, Problem Description (the new text box input).From Hospital/Doctor: Hospital Name, Doctor Name, Address.4. Automated Reminders (The 1-Day & 1-Hour Logic)We will use a Job Queue (recommended: BullMQ or node-cron) to poll for upcoming appointments.Check 1 (T-24h): Run every hour. Find appointments where appointment_time is between 24–25 hours away. Send SMS/Email.Check 2 (T-1h): Run every 15 minutes. Find appointments where appointment_time is between 60–75 minutes away. Send SMS/Email.5. API Endpoints (Phase 2)MethodEndpointDescriptionGET/api/v1/search?q=pancreasReturns hospitals/doctors matching keyword or name.GET/api/v1/availability/:doctorId?date=...Returns free/booked slots for a specific day.POST/api/v1/appointments/bookMandatory: doctor_id, patient_id, date, slot, problem_description.GET/api/v1/appointments/:id/cardGenerates the token card with all registry + booking details.6. Error Handling StrategyDouble Booking: If two patients try to book the same slot, the database UNIQUE constraint on (doctor_id, date, slot) will trigger an error, and the second user will get a message: "This slot was just taken. Please pick another."Search Miss: If a keyword isn't in our Symptom_Mapping, we fall back to a general hospital search or suggest "General Physician."Neon Connection: Since Neon is serverless, we'll handle "Cold Starts" with a retry logic in our Express middleware to ensure the user doesn't see a 500 error.
